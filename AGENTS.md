# AGENTS.md

## 项目范围

- 本仓库是基于 `ESP-IDF` 的 `ESP32-S3` 固件项目。
- 当前目标是围绕 `TEF6686` 构建一个 headless 调谐器设备。
- 用户可见的主路径应保持简单、稳定、可预期：
  - `USB CDC` 负责命令与状态交互
  - `TEF6686` 通过 `I2C` 控制
  - 优先保证 headless 运行，其次再考虑 UI 和周边工具

## 核心优先级

- 优先缩短从上电到可用调谐器的路径。
- 保持主机侧交互稳定、清晰、可预测。
- 把 `USB CDC`、`I2C` 和命令处理视为产品行为，而不只是内部实现细节。
- 优先选择适合当前固件阶段的简单方案，避免为了未来假设提前抽象。
- 改动保持最小化，并与现有代码风格一致。

## 仓库事实

- 顶层构建入口：`CMakeLists.txt`
- 主固件组件目录：`main/`
- 设计说明与补充文档：`docs/`
- 当前默认目标芯片：`esp32s3`

## 工作规则

- 修改前先阅读当前生效的 `AGENTS.md`。
- 新增逻辑前先搜索已有实现。
- 优先复用现有模块和接口。
- 不回退无关的用户修改。
- 非任务必要，不重命名文件、不移动模块、不新增抽象层。
- 依赖保持项目内声明，遵循现有项目配置方式。

## 固件实现约束

- 保持模块职责清晰：
  - `tef6686.*` 负责调谐器控制和调谐器相关数据结构
  - `usb_cdc.*` 负责 USB CDC 传输
  - `cmd_handler.*` 只负责命令解析和流程编排
- 硬件假设必须显式表达，不要藏在隐式状态里。
- 失败路径必须通过返回值或日志明确暴露。
- 避免跨模块隐藏状态变更。
- 优先稳定、可阻塞安全的实现，不追求技巧性并发。
- 对静态缓冲区、协议边界和任务时序保持谨慎。

## 代码风格

- 默认使用 ASCII。
- 代码注释仅使用英文。
- `import` 或 `#include` 放在文件顶部。
- 命名优先表达真实硬件语义或协议语义。
- 非必要不新增 flag 驱动的多模式函数。
- 注释只用于补充不直观的约束、硬件行为或协议意图。

## 错误处理

- 禁止静默失败。
- 能返回 `esp_err_t` 的地方优先返回具体错误。
- 日志要包含足够上下文，便于定位硬件和协议问题。
- 默认外部依赖都可能失败：
  - USB 连接状态可能变化
  - I2C 传输可能失败
  - 主机输入可能不合法

## 验证原则

- 改动后执行最小但有意义的验证。
- 对可构建的改动，优先执行 `idf.py build`。
- 对硬件相关改动，要明确区分“本地已验证”和“仍需上板验证”。
- 不默认新增测试。
- 只有在确实存在回归风险且仓库已有合适验证路径时，才补充测试。

## 常用命令

- `idf.py set-target esp32s3`
- `idf.py build`
- `idf.py flash`
- `idf.py monitor`

## 文档原则

- 文档应描述当前事实，而不是记录过程。
- 优先通过代码、命名和类型表达语义。
- 只有在行为、约束或工作流真实变化时才更新文档。

## 任务倾向

- 默认直接完成用户请求，不先做大范围重构提案。
- 存在多种实现方案时，优先选择对当前固件运行风险最低的方案。
- 优化目标是让设备更易控制、更易调试、更不容易被误操作。
